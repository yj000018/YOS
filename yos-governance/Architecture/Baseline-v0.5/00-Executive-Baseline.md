# 00 — Executive Baseline

## 1. Mission

Y-OS is a sovereign, modular, provider-agnostic cognitive operating system designed to preserve continuity, structure knowledge, route cognition, coordinate action, learn from outcomes and evolve its own capabilities under governance.

It is not merely:
- a chatbot;
- an agent framework;
- a memory layer;
- a tool registry;
- a workflow engine;
- a user interface.

It integrates all of these as parts of a living cognitive system.

## 3. Five-plane working canon

### Experience Plane
Human-facing interaction, supervision, presentation and project interfaces.

### Control Plane
Jobs, state, policy, orchestration, routing, context coordination, permissions, budgets and observability.

### Cognitive Plane
Perception, knowledge use, reasoning, intention, agency and action.

### Knowledge Plane
Raw sources, memory, provenance, structured knowledge, artifacts, context products and historical lineage.

### Growth Plane
Outcome evaluation, performance memory, capability evolution, route calibration, experimentation, pruning, promotion and rollback.

## 5. Core responsibilities

### Manus
Current primary human cockpit and program-direction environment.

### Y-Nexus
Future sovereign system boundary and control plane. Owns job identity, policies, permissions, budgets, observability and system state.

### Y-ORC
Operational orchestration runtime. Plans workflows, sequences missions and invokes CRT and ART.

### CRT
Chooses models, ensembles and cognitive strategies.

### ART
Chooses agents, workers, tools, APIs, MCP servers, runtimes, automations and effectors.

### Y-CTX
Determines which context is relevant.

### CCR
Compiles provider-ready context packs.

### PIE
Owns universal perceptual ingress and normalized observations.

### KAP
Owns provenance-aware knowledge acquisition, structuring, identity resolution and canonical knowledge publication.

### Reasoning
Owns inference, synthesis, judgment, planning and hypothesis formation. Final module name remains open.

### ACT
Owns effectuation/execution. Final acronym expansion and runtime boundary remain open.

### ARCH Collector
Preserves raw external sources, conversations, files, artifacts, attachments, versions and deltas.

### ARCH Archaeology
Reconstructs concept lineage, decisions, contradictions, naming changes and supersession.

### Y-REG
Registry of capabilities and activable system objects.

### Y-MEM
Historical memory umbrella covering episodic, semantic, procedural and continuity functions; current decomposition remains under review.

### Growth Plane
Uses outcomes and evaluation to improve capabilities and routing under explicit governance.

## 6. Anti-proliferation rules

1. A new name is not a new module.
2. An alias must remain an alias unless distinct ownership is proven.
3. A cross-plane function does not automatically require a new top-level service.
4. Rejected and superseded concepts remain in lineage records.
5. Existing modules are preferred when their scope can be clarified safely.
6. Every core capability must have exactly one primary owner.
7. Implementation status must be separated from conceptual status.
8. No canon promotion without source traceability.

## 7. Human/agent division of labor

- **ChatGPT:** architecture, synthesis, specifications, conceptual integration and canon candidates.
- **Manus:** current program cockpit, autonomous missions, web/files/deployment coordination and result presentation.
- **Codex:** default engineering farm for implementation, tests, repositories and runtime correctness.
- **Claude Code:** textual/architectural challenger, second engineer, reviewer and alternative implementer.
- **Native LLM interfaces:** only for measurable exclusive advantages; important work must be recoverable through ARCH.

## 8. Current maturity

The architecture is conceptually rich but operationally uneven:
- several modules are strongly defined;
- several implementation successes were reported;
- repository and runtime evidence has not yet been fully audited;
- some names and boundaries remain open;
- the five-plane model is a working canon, not final canon.
